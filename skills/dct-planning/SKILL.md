# 阶段三：计划编写

## 执行步骤

### 第一步：拆分需求到项目

分析上阶段产物（设计方案），识别需求涉及的所有项目，并拆分每个项目的职责边界。

**拆分原则：**
- 按仓库拆分：每个 Git 仓库独立

### 第二步：串行生成各项目计划

**根据项目拆分清单，按顺序为每个项目生成计划。**

不派遣子 Agent。sailor 自己使用 Read/Write/Edit 工具，按项目列表顺序逐个生成计划，必须严格按照计划要求生成计划。


**计划要求**
| # | 检查项 | 不满足则打回 |
|---|--------|--------------|
| 1 | **TDD 执行**：计划中每个步骤先写单元测试，再写实现代码 | 重写，从测试开始 |
| 2 | **测试覆盖**：单元测试必须覆盖边界情况（空值、异常、最大值等） | 补充测试后重审 |
| 3 | **设计与计划一致**：计划文件与阶段二审批通过的设计方案一致 | 重新对齐设计 |
| 4 | **checkpoint.md 逐条验收**：计划必须满足 prd/{date}-{需求}/checkpoint.md 每一条验收标准 | 补充计划内容 |
| 5 | **规范路由**：检测到项目类型后正确路由到对应规范文件 | 修正路由后重审 |
| 6 | **怀疑原则**：无证据不认为计划正确；边界情况必须有对应测试设计 | 补充证据 |
| 7 | **完整性检查**：检查计划是否按照每个项目进行拆分文件，并且检查有没有遗漏的项目 | 补充计划 |
| 7.1 | **需求覆盖检查**：每个 api.json 端点、每个数据库表、每个前端页面、每个 checkpoint 验收点都必须有对应任务 | 补充遗漏任务 |
| 8 | **保证字段真实**：需要使用 Mcp(mysql) 读取真实数据表字段，确保计划中使用的字段真实存在 | 重写计划 |
| 9 | **保证代码准确**：需要读取真实项目代码，决定模块新增还是修改，是否有可复用的代码 | 重写计划 |
| 10 | **前端项目必须做 UI/UX 设计**：如果项目是前端项目，必须调用 Skill(impeccable:frontend-design) 进行 UI/UX 设计 | 补充 UI/UX 设计后重审 |

**输出文件**
- `prd/{date}-{需求}/{project_name}-plan.md` — 每个项目单独一个计划文件


**每个项目的执行步骤：**

1. 调用 Skill(standards)，加载项目规范
2. **读取 design.md 和 api.json，提取本项目的设计点**
   - 提取本项目涉及的 api.json 端点
   - 提取本项目涉及的数据库表
   - 提取本项目涉及的前端页面
   - 提取本项目涉及的 checkpoint 验收点
3. **读取项目现有代码，提取可复用模式**（按需，不遍历全部）：
   - 先使用 `Grep` 搜索与需求主题相关的已有代码（如需求是"轮播图"，搜索 `banner|Banner`）
   - 如果找到已有实现，使用 `LSP goToDefinition` 读取关键函数/类型的定义
   - 如果无已有实现，搜索通用 CRUD 模式（如 `func.*List`、`func.*Create`）找到参考模块
   - 使用 `LSP documentSymbol` 浏览控制器/服务文件，只读取与当前需求**方法名相关**的方法体
   - **停止条件**：提取出 Controller/Service/Repository/Entity/DTO/Route 六层的代码书写模式后，立即停止，不读其他模块
4. 调用 Skill(superpowers:test-driven-development)，加载TDD规范
5. 调用 Skill(superpowers:writing-plans)，基于项目规范、设计点和代码模式生成计划
6. **生成需求覆盖矩阵**，验证每个设计点都有任务覆盖
7. 输出文件到 `prd/{date}-{需求}/{project_name}-plan.md`

### 第三步：检查产物并更新状态

1. 按项目列表逐个检查产物文件是否已生成：`{project_name}-plan.md`
2. 如有遗漏，重新生成
